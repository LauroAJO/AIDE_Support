INVESTIGAÇÃO (lógica de deduplicação encontrada ANTES da correção):

  1. POST /api/bridge/push/tasks → handleBridgePushTasks (_worker.js:3428)
     - Envia { tasks } com TODAS as tarefas do AIDE, mapeadas por shapeTask,
       para o Lifegame (POST /api/bridge/tasks).
     - shapeTask (_worker.js:724) inclui: id (o id DO AIDE), source ('aide' por
       padrão), lifegame_id, e os demais campos.
     - NÃO existia um campo dedicado `aideTaskId`. O id do AIDE ia só como `id`,
       que o Lifegame reatribui do lado dele.

  2. POST /api/bridge/receive/tasks → NÃO EXISTE.
     - O router (_worker.js:3198-3210) tem receive/sprints, receive/xp,
       receive/people, receive/timelog — mas NÃO receive/tasks.
     - A entrada de tarefas vindas do Lifegame acontece SÓ pelo caminho de PULL:
       handleBridgeImport(env, config, 'tasks') (rota /api/bridge/import/tasks,
       _worker.js:3205 / função :3711), que busca no LG e faz upsert.

  3. Deduplicação real → upsertLifegameTask (_worker.js:3656):
         const existing = SELECT id FROM tasks WHERE lifegame_id = ?  [bind t.id]
         if (existing) UPDATE title/urgency/importance/status/tags
         else INSERT novo com source='lifegame', lifegame_id = t.id
     - Casava APENAS por lifegame_id. Nenhum filtro por aideTaskId/source.

  ⇒ BUG CONFIRMADO: uma tarefa de origem AIDE (que tem lifegame_id = NULL) ao
    voltar do Lifegame nunca casa no WHERE lifegame_id = ? e é INSERIDA como nova
    tarefa source='lifegame'. A cada round-trip, duplicatas acumulam.


MUDANÇAS IMPLEMENTADAS (_worker.js):

  1. PUSH — handleBridgePushTasks (~linhas 3442-3450):
     Cada tarefa de ORIGEM AIDE é carimbada com aideTaskId (= id do AIDE) e
     source='aide' antes de enviar:
         const tasks = (results||[]).map(shapeTask).map((t) =>
           (t.source||'aide') !== 'lifegame' ? { ...t, aideTaskId: t.id, source: 'aide' } : t);
     Tarefas de origem 'lifegame' seguem SEM aideTaskId, para continuarem
     sincronizando normalmente pelo lifegame_id no retorno (sem regressão).

  2. IMPORT — handleBridgeImport, loop de itens (~linhas 3743-3783):
     Passo 1 (filtro anti-eco, só entity==='tasks'):
         if (item.aideTaskId || item.source === 'aide') { skip + log; continue; }
       → conta skippedEcho e registra no bridge_sync_log:
         { direction:'skipped_echo', entity_type:'task', status:'skipped',
           entity_id: aideTaskId|id,
           payload: JSON.stringify({ reason:'echo_loop_prevention', aideTaskId }) }
     Passo 2 (tarefas externas restantes): segue o upsert existente
       (upsertLifegameTask), que já deduplica por lifegame_id (INSERT/UPDATE).
     Resposta e log de resumo agora incluem `skipped_echo` / "eco ignorados".

  Observação de projeto: a dedup por lifegame_id foi MANTIDA (upsertLifegameTask
  inalterado). O campo lifegame_id já é o "id próprio do Lifegame" e é indexado
  por idx_tasks_lifegame_id.
