     - Ao clicar "Salvar" (função save, linhas ~200-232 do original): montava um
       array `ops` e disparava UMA requisição HTTP POR checkbox (~48 no total) com
       Promise.all:
          * se a caixa == preset  → DELETE (com .catch(()=>null), erro engolido);
          * se difere do preset    → PUT { feature, action, allowed }.
       => Dezenas de requests concorrentes ao mesmo endpoint, cada um com uma
          escrita no D1. Sob concorrência, algumas escritas falhavam/racejavam; os
          DELETEs eram silenciados e o Promise.all só propaga a 1ª rejeição → certas
          permissões (ex.: tasks.view_all/create, notes.view_own/create) não
          persistiam, sem erro visível. ESSE é o bug.

  2. _worker.js — PUT /api/users/:id/granular-permissions (handleUserGranular-
     Permissions, ~linha 7057):
     - PUT aceitava APENAS um item único { feature, action, allowed } e fazia
       INSERT ... ON CONFLICT(user_id, feature, action) DO UPDATE (linhas 7084+).
       Não havia modo em lote. Combinado com o modal disparando dezenas de PUT/
       DELETE únicos e concorrentes, era o caminho frágil.
     - Sem validação que "derrube" features específicas — o problema era a
       concorrência, não a validação.
     - GET/DELETE existiam; DELETE só removia 1 override por vez.

  ⇒ LOCAL DO BUG: PermissionsModal.save() (fan-out de ~48 requests concorrentes,
    com DELETEs silenciados) + PUT do worker aceitando só item único (sem lote
    atômico). Corrige-se batendo tudo numa requisição/transação única.

PENDÊNCIAS / DESVIOS:

      spec). Nenhuma alteração necessária ali.
  D2) MODELO DE SALVAMENTO mudou de "diff vs preset" (só overrides que diferem)
      para "todas as caixas explícitas". Agora cada caixa salva vira um override

      a herdar do preset. Comportamento efetivo de permissão é idêntico.

      ser visível (não há sistema de toast global).
